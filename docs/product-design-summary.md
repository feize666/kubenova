# KubeNova 完整产品方案与设计总结

## 1. 项目概述

KubeNova 是一个面向 Kubernetes 多集群场景的 AIOps 运维控制面。项目目标不是单点资源 CRUD，而是建立一个覆盖“资源管理、关系观察、运行时接入、权限控制、AI 辅助、部署交付”的完整平台。

项目由三层组成：

- `frontend`
  基于 Next.js + React + Ant Design 的统一控制台
- `backend/control-api`
  基于 NestJS + Prisma + PostgreSQL + Redis 的业务控制面 API
- `backend/runtime-gateway`
  基于 Go 的运行时网关，负责终端、日志、实时会话与静态资源承载

这是一个前后端一体的运维平台，不是单纯前端展示项目。

## 2. 产品定位

产品定位为：

**面向平台管理员、SRE、运维工程师和交付团队的 Kubernetes 多集群 AIOps 控制面。**

平台核心价值：

- 统一多集群入口
- 统一资源管理入口
- 统一关系可视化入口
- 统一运行时入口
- 统一 AI 辅助入口

## 3. 产品目标

### 3.1 业务目标

- 降低多集群运维复杂度
- 缩短从发现问题到执行动作的路径
- 把高频资源管理从 YAML/CLI 拉回可治理控制面
- 为 AI 辅助诊断和自动化动作提供结构化上下文

### 3.2 平台目标

- 建立资源、关系、运行态统一模型
- 提供清晰的资源域导航与信息架构
- 支撑后续告警、巡检、资源图谱、AIOps 闭环联动

## 4. 目标用户

### 平台管理员

- 管理集群接入、用户权限、系统更新、平台能力开关
- 关注平台健康、合规、安全、可用性

### SRE / 运维工程师

- 管理工作负载、网络、存储、配置
- 查看资源关系、排查异常、进入终端、跟踪日志

### 交付 / 应用团队

- 在限定名称空间内查看服务、部署、入口与配置
- 完成常规变更和问题自助定位

## 5. 整体系统架构

## 5.1 前端层

前端承担：

- 导航信息架构呈现
- 资源列表、资源详情、YAML 编辑
- 资源全景图和可视化工作台
- 用户认证后的交互控制
- 与控制面 API 和运行时网关联动

当前前端设计原则：

- 以资源域为骨架组织界面
- 高频操作前置
- 详情与列表并重
- 图谱页采用筛选优先、画布优先的工作台模式

## 5.2 控制面 API

`control-api` 是系统主业务中枢，承担：

- JWT 认证与会话管理
- 用户、RBAC、审计
- 集群注册与凭据管理
- 工作负载 / 网络 / 存储 / 配置资源的查询与操作
- 数据持久化
- 资源同步、状态聚合、健康评估
- AI 助手动作编排入口

它是系统的治理平面，不直接承担长连接运行时细节。

## 5.3 运行时网关

`runtime-gateway` 承担：

- 终端会话 WebSocket
- 日志流 WebSocket
- 运行时交互中转
- 健康探针与版本接口
- 静态资源 fallback 承载

设计上将“业务控制”和“运行时流量”解耦，避免 control-api 同时承担短请求与长连接压力。

## 5.4 数据层

当前数据基础：

- PostgreSQL
  业务主存储、用户、会话、资源记录、能力快照、审计等
- Prisma
  数据模型与 ORM
- Redis
  队列、缓存、异步处理能力支撑

当前数据库设计已经把 Kubernetes 资源抽象为多个治理域：

- `WorkloadRecord`
- `NetworkResource`
- `StorageResource`
- `ConfigResource`
- `NamespaceRecord`
- `ApiResourceCapability`
- `ClusterHealthSnapshot`

这意味着平台不是只查实时 Kubernetes API，而是在做“控制面缓存 + 治理模型”。

## 6. 功能域划分

## 6.1 总览

总览承担平台入口能力：

- 风险感知
- 资源概览
- 快捷跳转
- 全局态势

## 6.2 资源全景图

资源全景图是图谱工作台，不是装饰页。

目标：

- 查看资源关系
- 切换集群与名称空间
- 聚焦资源类型
- 从入口到控制器到 Pod 看完整链路
- 为故障定位与 AI 分析提供结构化视觉上下文

## 6.3 集群管理

集群管理承担：

- 集群注册
- kubeconfig 管理
- 健康检测
- 同步能力与可读性校验

## 6.4 网络管理

网络域设计参考 Headlamp，并结合本项目治理需求增强。

当前网络域资源：

- Service
- Endpoints
- EndpointSlice
- Ingress
- IngressRoute

网络域设计原则：

- `Service` 管暴露与寻址
- `Endpoints / EndpointSlice` 管真实后端目标
- `Ingress / IngressRoute` 管入口流量规则

从而形成：

**入口 -> 服务 -> 后端目标**

的完整观察路径。

## 6.5 工作负载

工作负载域覆盖：

- Pod
- Deployment
- StatefulSet
- DaemonSet
- ReplicaSet
- Job
- CronJob
- 镜像仓库
- Namespace

目标不是只做“列资源”，而是支持：

- 发布观察
- 副本状态判断
- 关联网络和存储
- 进入运行时

## 6.6 弹性伸缩

- HPA
- VPA

是从资源控制过渡到策略控制的能力域。

## 6.7 存储管理

- PV
- PVC
- StorageClass

与工作负载和配置管理形成联动。

## 6.8 配置管理

- ConfigMap
- Secret
- ServiceAccount

承担应用配置与身份配置入口。

## 6.9 身份与安全

- 用户管理
- RBAC
- 安全管理

为平台的多角色治理提供基础能力。

## 6.10 可观测性

- 集群健康
- 日志中心
- 资源巡检

后续应进一步与资源图谱和 AI 助手打通。

## 6.11 AIOps 中台

AIOps 不应只是聊天窗口，而应是平台增强层。

目标方向：

- 基于当前页面上下文理解用户意图
- 读取资源与关系上下文
- 生成建议
- 编排动作
- 记录审计

## 6.12 系统管理

- 更新管理
- 平台运行级配置

## 8. 前端设计原则

### 8.1 资源列表页

资源列表页应遵循：

- 筛选优先
- 列信息克制但可执行
- 上方动作栏统一
- 详情、YAML、删除等高频动作直接可达

### 8.2 资源详情页

资源详情页应统一为：

- 概览
- 运行态
- 关联关系
- 网络
- 存储
- 事件
- 元数据

这套抽象已经在前后端资源详情中初步形成。

### 8.3 图谱页

图谱页设计基线：

- 先筛选，后观察
- 默认画布优先
- 详情按需展开
- 资源类型聚焦统一入口
- 名称空间筛选是一等控件
- 分组控件应接近 Headlamp 的图谱工具思路

### 8.4 视觉方向

- 企业级浅色控制台
- 高信息密度但不过载
- 一致的圆角、轻阴影、低饱和边框
- 用 chip / pill 表达筛选和状态

## 9. 后端设计原则

### 9.1 控制面优先

后端不应把前端变成“直接调 Kubernetes API 的薄壳”。

当前正确方向是：

- 通过 control-api 建立统一业务语义
- 在数据库里持久化资源记录与快照
- 对外输出统一 contract

### 9.2 资源域模型

资源记录按治理域拆分，而不是一个万能表：

- workload
- network
- storage
- config

这样更利于：

- 领域内操作封装
- 详情页抽象
- 后续扩展 capability / sync / graph / AI

### 9.3 运行时解耦

`runtime-gateway` 与 `control-api` 拆开，是正确架构。

这样可以：

- 把 WebSocket 和 API 请求压力拆开
- 单独演进终端、日志、实时流能力
- 让安全边界更清晰

### 9.4 集群同步机制

系统通过同步将 Kubernetes 资源写入平台模型，这是关键基础能力。

同步价值：

- 支持统一检索
- 支持资源图谱
- 支持详情与关系计算
- 支持后续告警 / AI / 巡检联动

## 10. 网络域当前设计总结

这次网络域已经从“Service / Ingress 两页”扩展为完整链路。

新增与增强包括：

- IngressRoute 页面与后端链路
- Endpoints 页面
- EndpointSlice 页面
- NetworkResource 类型扩展
- 集群同步支持 Endpoints / EndpointSlice
- 资源详情对新类型的识别

这让网络域真正形成了管理闭环。

## 11. 资源全景图当前设计总结

资源全景图当前已从普通拓扑页转成图谱工作台方向。

已形成的原则：

- 集群筛选
- 名称空间筛选
- 资源类型聚焦
- 视角切换
- 默认画布优先
- 详情侧栏按需展开

明确去掉的冗余设计：

- 名称空间分组
- 重复的资源类型切换与聚焦双入口
- 占空间但低价值的画布说明标题

## 12. 部署与运行模型

项目已经具备多种部署形态：

- 本地开发启动脚本
- Docker Compose
- K8s Kustomize
- systemd / 二进制
- deb/rpm 打包

说明项目方向不是 Demo，而是具备交付与运行基础的产品雏形。

## 13. 当前能力成熟度判断

### 已相对稳定

- 登录认证
- 资源列表骨架
- 多类资源管理页
- 网络域基础闭环
- 构建与本地开发链路

### 正在形成平台壁垒

- 资源全景图
- AIOps 助手与动作入口
- 资源详情统一抽象
- 集群同步与能力快照

### 仍需继续建设

- 图谱与告警联动
- Service -> Endpoints -> EndpointSlice 深度联动
- NetworkPolicy / Gateway API
- 更完整的审计与动作回放
- AI 助手与具体资源上下文的双向绑定

## 14. 后续路线建议

### 短期

1. 完善 Service 详情对 Endpoints / EndpointSlice 的直接联动
2. 继续压缩资源全景图控件区，提升画布效率
3. 统一网络资源页摘要字段与操作体验
4. 打通日志、终端、详情页间跳转

### 中期

1. 引入 NetworkPolicy / Gateway API
2. 将资源图谱从页面实现升级为模块化 graph pipeline
3. 打通巡检、告警、图谱、AI 助手
4. 建立资源动作审计闭环

### 长期

1. 形成“资源状态 -> 关系图谱 -> AI 建议 -> 运维动作”的完整闭环
2. 建立多集群、多租户、角色隔离下的企业级运维平台能力
3. 让平台从“资源控制台”升级为“智能运维中台”

## 15. 最终结论

这个项目的正确方向，不是继续堆更多资源列表，而是：

**以前端统一控制台为入口，以 control-api 作为治理中枢，以 runtime-gateway 作为实时能力底座，以资源关系图谱和 AIOps 作为平台增强层，逐步演进为 Kubernetes 智能运维控制面。**

后续开发应持续围绕四件事推进：

1. 资源域信息架构更清晰
2. 资源关系更可见
3. 控制面与运行态联动更顺
4. AI 能力真正接入动作闭环
